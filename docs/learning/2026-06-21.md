# 2026-06-21 - Publication State and Ownership

## Goal

Day 5 focuses on publication state and ownership: a pipeline can write a table, but consumers should only bind to a version that has passed ownership, version, and quality gates.

The goal is to model publication as a control-plane decision:

- A current published version exists.
- Candidate tables can be written safely but are not consumer-visible yet.
- Publication requests carry requester, owner, expected version, proposed version, and target table.
- Quality evidence and structural checks decide whether a request is ready to publish or blocked.
- Only the approved request updates the stable published table and product registry.

## What I Learned

Publication is the visibility boundary. A write creates data, but publication decides whether consumers should use it.

The key distinction:

```text
Candidate table exists      -> engineering work has produced data
Published version exists    -> platform has approved consumer visibility
```

Day 5 uses four requests:

```text
pub-001: owner, expected version 4, good candidate       -> READY_TO_PUBLISH
pub-002: non-owner, expected version 4, good candidate   -> BLOCKED
pub-003: owner, stale expected version 3, good candidate -> BLOCKED
pub-004: owner, expected version 4, bad candidate        -> BLOCKED
```

This models three production controls:

- Ownership: only the accountable owner can publish the product.
- Expected version: prevents lost updates and stale approvals.
- Quality evidence: proves the candidate is safe enough for publication.

The important operational point is that rejected candidates still remain inspectable. They are not thrown away. They become evidence for debugging, review, and improvement.

Principal-level point: mature data platforms separate compute success from publication success. A job can finish, write files, and pass basic schema checks while still being unsafe to publish.

## PySpark Notes

The notebook uses PySpark to evaluate publication gates because the decision combines registry metadata with per-candidate quality checks.

SQL equivalent for the structural checks:

```sql
SELECT
  r.*,
  p.owner_email,
  p.current_version,
  r.requested_by = p.owner_email AS owner_ok,
  r.expected_current_version = p.current_version AS expected_version_ok,
  r.proposed_version = p.current_version + 1 AS version_increment_ok
FROM publication_requests_delta_day5 r
JOIN data_products_delta_day5 p
  ON r.product_id = p.product_id;
```

PySpark shape:

```python
requests_df = spark.table("de_learning.publication_requests_delta_day5")
products_df = spark.table("de_learning.data_products_delta_day5")

request_context_df = (
    requests_df
    .join(products_df, on="product_id", how="left")
    .withColumn("owner_ok", F.col("requested_by") == F.col("owner_email"))
    .withColumn("expected_version_ok", F.col("expected_current_version") == F.col("current_version"))
    .withColumn("version_increment_ok", F.col("proposed_version") == F.col("current_version") + F.lit(1))
)
```

Important syntax:

- `spark.table(...)` reads SQL tables into DataFrames.
- `.join(..., on="product_id", how="left")` is the PySpark form of a SQL left join.
- `.withColumn(...)` adds derived check columns.
- `F.col("requested_by") == F.col("owner_email")` compares two DataFrame columns.
- `F.lit(1)` creates a literal value so `current_version + 1` is expressed as a column operation.
- `createOrReplaceTempView(...)` lets later SQL cells persist PySpark decisions.

The notebook also uses a small `collect()` loop to inspect candidate tables by name. That is acceptable for this tiny lab. In production, candidate validation should run through a scalable validation service or generated SQL plan.

## Mistakes / Confusions

- A candidate table existing does not mean it is published.
- A successful job does not mean a successful data product release.
- A non-owner can produce a technically valid candidate, but ownership still matters for publication.
- Expected-version checks are not ceremony; they prevent stale publication decisions and lost updates.
- Quality failures should produce evidence, not just failed jobs.

## Revision Cards

Q: What is the difference between writing a table and publishing a table?
A: Writing creates or updates data. Publishing makes a version consumer-visible after ownership, version, and quality checks pass.

Q: Why should consumers bind only to published versions?
A: Candidate tables may be incomplete, unapproved, stale, or failing quality checks. Published state is the stable contract boundary.

Q: What production bug does `expected_current_version` prevent?
A: Lost updates. A request made against version 3 should not publish over version 4 without explicitly revalidating.

Q: Why is owner approval required if quality checks pass?
A: Quality checks prove technical evidence. The owner is accountable for the product's semantics, consumer impact, and release intent.

Q: Why keep blocked requests and evidence?
A: They explain why a change was rejected and provide audit/debugging material for the owner and platform team.

Q: What does publication give beyond Delta table history?
A: Delta history tells what happened to a table. Publication records tell whether a version was approved for consumers, by whom, against which checks and expected version.
