## What I Learned

Governance has to separate internal storage from consumer surfaces.

In this lab:

- `customers_secure_base_day11` is the internal restricted PII table.
- `customers_masked_view_day11` is the consumer-facing governed view.
- `consumer_context_day11` simulates role and regional access.
- `access_decisions_delta_day11` records access decisions and reasons.

The core design pattern:

```text
restricted base table -> governed view -> access decision evidence -> audit summary
```

Unity Catalog implements this with catalogs, schemas, grants, row filters, column masks, volumes, and ownership. The lab models the same mental model with runnable Delta objects so the workflow can be practiced in a personal workspace.

Principal-level point: governance is not a PDF policy. It is a set of enforceable, queryable controls that protect data while still allowing useful consumer access.

## PySpark Notes

The notebook uses PySpark to evaluate access requests.

```python
access_decision_df = (
    requests_df
    .join(objects_df, requests_df.requested_object == objects_df.object_name, "left")
    .join(context_df, on="principal", how="left")
    .withColumn("object_exists", F.col("object_type").isNotNull())
    .withColumn("allowed", ...)
    .withColumn("decision", F.when(F.col("allowed"), F.lit("APPROVED")).otherwise(F.lit("DENIED")))
)
```

SQL equivalent:

```sql
SELECT r.*, o.object_type, c.role_name,
  CASE
    WHEN r.requested_object = 'customers_masked_view_day11' THEN 'APPROVED'
    WHEN r.requested_object = 'customers_secure_base_day11' AND c.role_name = 'privacy_admin' THEN 'APPROVED'
    ELSE 'DENIED'
  END AS decision
FROM access_requests_day11 r
LEFT JOIN uc_objects_day11 o ON r.requested_object = o.object_name
LEFT JOIN consumer_context_day11 c ON r.principal = c.principal;
```

Important syntax:

- `join(..., how="left")` keeps every request even if metadata is missing.
- `withColumn(...)` adds policy-derived columns.
- `F.when(...).otherwise(...)` is SQL `CASE WHEN`.
- Boolean expressions with `&` and `|` model policy rules.
- `createOrReplaceTempView(...)` bridges the PySpark result back to SQL.

## Mistakes / Confusions

- A table containing PII should not automatically be the consumer-facing table.
- Masking and row filtering are different controls: one changes visible values, the other changes visible rows.
- A direct base-table access request can be denied even when a masked view access request is approved.
- Governance decisions need reasons, not just approved/denied flags.
- A personal workspace lab may simulate Unity Catalog controls, but the production design should use actual Unity Catalog grants, row filters, masks, and audit logs.

## Revision Cards

Q: What is the safe consumer surface in this lab?
A: `customers_masked_view_day11`, not the restricted `customers_secure_base_day11` table.

Q: What does row filtering do?
A: It restricts which rows a consumer can see, such as APAC-only rows for an APAC analyst.

Q: What does masking do?
A: It changes sensitive column values, such as replacing real email with a masked email.

Q: Why should base PII table access be denied to analysts?
A: Analysts can often do their work through a governed view without direct raw PII access.

Q: Why is `privacy_admin` allowed to access the base table?
A: The role has a privacy review purpose that justifies direct PII access.

Q: What should an access decision record contain?
A: Principal, role, object, privilege, decision, and reason.

Q: What is the Professional-level habit in this lab?
A: Turn governance rules into auditable tables and reviewable evidence.
