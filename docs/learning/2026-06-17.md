## What I Learned

A data contract is not just schema. Schema says what fields exist. A platform contract says who owns the data, what one row means, which keys identify a record, what freshness is promised, how sensitive the data is, and what changes are compatible.

The principal-level design move is to make compatibility machine-checkable before data reaches downstream consumers. That does not remove human review; it gives reviewers concrete evidence.

Concrete example:

```text
Safe: add nullable coupon_code
Breaking: remove customer_id
Breaking: change amount from decimal(12,2) to string
Breaking: remove order_id from the primary key
```

## Mistakes / Confusions

- The expected companion repo path did not exist, so the first step was creating a minimal repo shell rather than extending existing code.
- It is easy to under-model the contract as only column names and types. The important fields are owner, grain, key, freshness, sensitivity, and lifecycle status.
- Java records generate accessors with component names. Static factory names should not collide with component accessors.

## Revision Cards

Q: What is the difference between schema and data contract?
A: Schema describes field shape. A data contract includes schema plus ownership, grain, keys, freshness, sensitivity, lifecycle, and compatibility rules.

Q: Why does grain belong in the contract?
A: Consumers need to know what one row represents. A table with `order_id` can mean one row per order, per order line, per order event, or per daily order aggregate.

Q: Why is removing a column breaking even if current dashboards do not use it?
A: The platform cannot assume current observed usage is complete. A published contract is a consumer API unless deprecation and lineage impact are handled explicitly.

Q: What is the production failure this checkpoint prevents?
A: Silent producer changes that keep pipelines green while downstream answers drift or break.
