# 2026-06-23 - Replay And Backfill Safety

## Goal

Day 7 focuses on replay and backfill safety.

The goal is to make daily reruns predictable:

- A naive append rerun duplicates aggregate rows.
- A deterministic input fingerprint separates same-input replay from changed-input rerun.
- A run registry records which batch/run was attempted and whether it was allowed.
- An idempotent `MERGE` publishes by output grain key instead of appending duplicates.
- A corrected backfill uses a new batch id instead of reusing the old one.

## What I Learned

Daily pipelines usually fail in boring ways: retries, partial reruns, and backfills. The dangerous case is not that a job fails visibly. The dangerous case is that the same input is appended twice and a dashboard shows doubled revenue.

The key distinction:

```text
append publication -> every rerun creates more rows
merge publication  -> every rerun converges to the same grain key
```

For gold metrics, the output grain is often a date plus dimensions. In Day 7 the grain is just `metric_date`. Publishing with `MERGE ON metric_date` means the same day is updated in place instead of appended again.

But idempotent output is not enough. The platform also needs to know whether the input was the same. That is why the lab computes an `input_fingerprint` from deterministic row tokens. If the same batch id produces a different fingerprint, the platform blocks the publish. That is not a compute failure; it is a control-plane safety decision.

Corrected backfills are still allowed. They should use a new batch id, record a new run, and then update the published table with the corrected values.

Principal-level point: a replay-safe pipeline has both data-plane idempotency and control-plane evidence.

## PySpark Notes

The notebook uses PySpark to build a deterministic candidate and run context.

```python
source_df = spark.table("de_learning.orders_silver_snapshot_day7")

batch_input_df = source_df.where(
    (F.col("order_date") >= F.to_date(F.lit(start_date)))
    & (F.col("order_date") <= F.to_date(F.lit(end_date)))
)
```

SQL equivalent:

```sql
SELECT *
FROM orders_silver_snapshot_day7
WHERE order_date BETWEEN DATE '2026-06-01' AND DATE '2026-06-02';
```

Candidate metric:

```python
candidate_df = (
    batch_input_df
    .groupBy("order_date")
    .agg(
        F.count("*").alias("order_count"),
        F.sum(
            F.when(F.col("current_status") == "completed", F.col("current_amount"))
             .otherwise(F.lit(0).cast("decimal(10,2)"))
        ).alias("completed_revenue"),
    )
)
```

SQL equivalent:

```sql
SELECT order_date, COUNT(*), SUM(CASE WHEN current_status = 'completed' THEN current_amount ELSE 0 END)
FROM orders_silver_snapshot_day7
GROUP BY order_date;
```

Important syntax:

- `spark.table(...)` reads a SQL table into a DataFrame.
- `where(...)` is SQL `WHERE`.
- `groupBy(...).agg(...)` is SQL `GROUP BY`.
- `F.when(...).otherwise(...)` is SQL `CASE WHEN`.
- `createOrReplaceTempView(...)` exposes the DataFrame to later SQL cells.
- DataFrames are lazy until an action such as `display(...)`, a write, or a collect-like operation.

## Mistakes / Confusions

- A successful `CREATE TABLE AS SELECT` cell may show write metadata instead of table rows. Verify with a separate `SELECT`.
- An aggregate table can be syntactically valid and operationally wrong if the same metric date appears twice.
- `batch_id` should not be reused for changed input. A changed input snapshot needs a new batch identity.
- `MERGE` prevents duplicate output rows, but the run registry explains whether the replay was allowed.
- Delta history shows table operations; the run registry explains intent and safety decisions.

## Revision Cards

Q: Why is append unsafe for rerunnable aggregate jobs?
A: A retry inserts another copy of the same aggregate rows, so downstream sums can double-count.

Q: What key makes the Day 7 gold publish idempotent?
A: `metric_date`, because the target grain is one row per metric date.

Q: What does the input fingerprint protect against?
A: Reusing the same batch id for a different input snapshot.

Q: Why is a corrected backfill not published under the old batch id?
A: The old batch id already means a specific input fingerprint. Reusing it would destroy auditability.

Q: What is the difference between `run_id` and `batch_id`?
A: `run_id` identifies an execution attempt. `batch_id` identifies the logical input window/snapshot being published.

Q: What does `MERGE` give beyond `INSERT INTO`?
A: It updates or inserts by a deterministic key, so replay converges instead of appending duplicates.

Q: What should a production backfill record?
A: Input range, input snapshot/fingerprint, code version, contract version, quality result, owner/approver, output target, and decision.
