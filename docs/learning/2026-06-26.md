## What I Learned

Schema evolution has two separate questions:

```text
Can the storage table accept the schema change?
Can consumers safely read the changed contract?
```

Delta can record table operations and schema metadata, but it does not automatically know whether a change is safe for every consumer. That decision belongs to the data contract and publication gate.

Adding a nullable column is usually backward-compatible:

- Existing rows can have `NULL`.
- Old readers can ignore the new field.
- New readers can start using it when ready.

Changing the type of an existing required column is usually incompatible. Removing a required column is also incompatible. A candidate table with those changes can exist physically, but it should not be published as the stable product.

Principal-level point: compatibility is an operational contract, not just a Spark/Delta capability. `DESCRIBE HISTORY` answers what happened. Compatibility evidence answers whether the change was safe.

## PySpark Notes

The notebook uses PySpark to read table schemas and compare them to the contract registry.

```python
fields = spark.table(f"de_learning.{table_name}").schema.fields
```

SQL equivalent concept:

```sql
DESCRIBE table_name;
```

The compatibility check joins expected contract columns to actual schema columns:

```python
contract_df.join(actual_schema_df, on="column_name", how="full_outer")
```

SQL equivalent:

```sql
SELECT *
FROM expected_contract
FULL OUTER JOIN actual_schema
  ON expected_contract.column_name = actual_schema.column_name;
```

Important syntax:

- `spark.table(...).schema.fields` reads schema metadata rather than table rows.
- `spark.createDataFrame(...)` converts Python `Row` objects into a Spark DataFrame.
- `join(..., how="full_outer")` catches missing and extra columns.
- `F.when(...).otherwise(...)` is SQL `CASE WHEN`.
- `createOrReplaceTempView(...)` bridges the PySpark evidence into SQL publication gates.

## Mistakes / Confusions

- A table can physically exist and still be unsafe to publish.
- Adding a nullable column is different from changing the type of an existing column.
- Required-column removal should be blocked even if a downstream query can be rewritten.
- `DESCRIBE HISTORY` does not replace compatibility evidence.
- Optional columns can be absent in some compatibility policies, but required columns cannot.

## Revision Cards

Q: Why is adding a nullable column usually backward-compatible?
A: Existing rows can store `NULL`, old readers can ignore the field, and new readers can opt in.

Q: Why is changing `amount` from `decimal(10,2)` to `string` unsafe?
A: Consumers expecting numeric arithmetic may break or produce wrong results.

Q: Why is removing `status` blocked?
A: It is a required lifecycle column in the contract.

Q: What does `DESCRIBE HISTORY` tell you here?
A: It shows table operations such as create, insert, and alter, but not whether the change was contract-compatible.

Q: What does schema compatibility evidence add?
A: It records expected type, actual type, required status, outcome, and publishability.

Q: Why use a candidate table before publishing?
A: It lets the platform validate schema and quality before changing the stable consumer-facing table.

Q: What should a publication gate block?
A: Required-column removal, incompatible type changes, unsafe nullability changes, and unapproved extra columns when policy disallows them.
