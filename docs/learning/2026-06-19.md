# 2026-06-19 - Ownership and Contract Publication State

## Goal

Phase 1, checkpoint CP3: add ownership and approval state transitions around source contract changes.

The point is to model contract publication as a controlled state transition, not a direct edit to a table definition. Consumers should bind to published versions, while proposed and validated versions remain reviewable control-plane state.

## Build Work

Extended the `data-platform-contracts` module.

Implemented:

- `ContractStatus.VALIDATED` - intermediate status between `PROPOSED` and `ACTIVE`.
- `SourceContract.withStatus(...)` - immutable copy helper for state transitions.
- `ContractTransition` - audit record for request id, from/to status, actor, decision time, and reason.
- `SourceContractChange` - immutable state machine around a proposed source contract version.

The allowed lifecycle is:

```text
PROPOSED -> VALIDATED -> ACTIVE -> DEPRECATED -> RETIRED
```

The model enforces:

- Only the source owner can propose, validate, activate, deprecate, or retire the contract.
- Validation checks that the new version is exactly `expectedPreviousVersion + 1`.
- Validation refuses backward-incompatible schema/key changes.
- Retirement cannot skip deprecation.

## What I Learned

Publication is the important boundary. A producer can draft a contract, validation can prove compatibility, and the owner can approve it, but consumers should only depend on the active version.

This is the same principle as a deployment pipeline: code can be in a branch, built, tested, and approved before production traffic sees it. For data contracts, the states make the platform answer:

```text
Who proposed the change?
Who approved it?
Which previous version did it expect?
Was compatibility checked?
When did consumers see it?
Was it deprecated before retirement?
```

Without this state machine, a schema registry or catalog becomes a mutable document store. With it, contract publication becomes auditable platform behavior.

## Mistakes / Confusions

- `ACTIVE` alone is not enough. A platform also needs `PROPOSED` and `VALIDATED` so CI and owner approval can happen before publication.
- Owner approval is not a substitute for compatibility checks. The owner can approve intent, but the platform still has to reject unsafe transitions.
- Deprecation and retirement are different. Deprecation starts a migration window; retirement means consumers should no longer bind to that contract.

## Revision Cards

Q: Why should contract publication be a state transition?
A: It prevents half-visible changes and gives the platform an audit trail for who approved what, against which prior version, and when consumers could see it.

Q: Why is `VALIDATED` separate from `ACTIVE`?
A: Validation proves the change passed checks. Activation publishes it to consumers. Keeping them separate lets CI, preview, approval, and publication be reasoned about independently.

Q: What production bug does `expectedPreviousVersion` prevent?
A: Lost updates. Two owners or jobs cannot unknowingly publish changes against stale contract state.

Q: Why cannot a contract go directly from `ACTIVE` to `RETIRED`?
A: Consumers need a deprecation window. Direct retirement creates abrupt breakage and hides migration responsibility.

Q: Who is allowed to approve the source contract in this checkpoint?
A: The source owner named in the contract. Later checkpoints can add delegated approvers and policy-specific reviewers.
