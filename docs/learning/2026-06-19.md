## What I Learned

A bronze table should preserve source fidelity, but "raw" does not mean "uncontrolled." A useful bronze ingestion path still needs enough metadata to make future debugging possible.

In this checkpoint, each accepted event carries:

- `event_id` - idempotency key for duplicate delivery.
- `source_lsn` - source ordering/replay position.
- `operation` - CDC operation type.
- `contract_id` and `contract_version` - the contract that shaped the row.
- `source_system` - the system of record.
- `occurred_at` and `ingested_at` - source time versus platform ingestion time.

The important distinction is that `order_id` is the business entity key, while `event_id` is the delivery/event key. If an order changes five times, `order_id` repeats five times. If the connector retries the same delivery, `event_id` repeats for the same event. That is why `event_id` is the right idempotency key for this bronze event table.

Rejected rows should not silently disappear. The quarantine table keeps bad rows with `dq_error`, which gives the platform a repair path:

```text
missing order_id
negative amount
invalid operation
wrong contract_version
```

Principal-level point: data quality should create evidence, not just failures. A failed row with a reason is operationally useful. A dropped row is a future incident.

## Mistakes / Confusions

- Bronze is raw, but it still needs metadata. Otherwise replay, debugging, lineage, and incident response become guesswork.
- `order_id` and `event_id` solve different problems. `order_id` identifies the business entity; `event_id` identifies a specific source change delivery.
- A row-count check is useful but incomplete. The expected result is `3` bronze rows and `2` quarantine rows, but correctness also requires checking duplicates, metadata, and quarantine reasons.
- `CREATE OR REPLACE TABLE` makes the lab deterministic. In production, replacement is a publication decision and should not happen casually.

## Revision Cards

Q: Why is `event_id` better than `order_id` for idempotent bronze event ingestion?
A: `event_id` identifies a source event delivery. `order_id` identifies the business entity and can appear in many create/update/delete events.

Q: Why should bad rows go to quarantine instead of being dropped?
A: Quarantine preserves evidence and enables repair. Dropping rows hides data loss and makes downstream discrepancies harder to explain.

Q: What does `contract_version` help debug later?
A: It tells you which contract shaped the row when it entered the platform, so schema/semantic changes can be correlated with downstream behavior.

Q: Why is bronze allowed to be messy but not unexplained?
A: Bronze preserves source fidelity, including imperfect inputs, but it must still carry enough metadata to support replay, lineage, quality checks, and incident response.

Q: What should happen if this same batch is rerun?
A: `orders_delta_day3` should not gain duplicate events because the merge uses `event_id` as the idempotency key. Quarantine should also avoid duplicate rejected rows by matching `event_id` and `dq_error`.
