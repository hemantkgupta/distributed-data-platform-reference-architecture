# 2026-06-25 - Delete Semantics And Tombstones

## Goal

Day 8 focuses on delete semantics and tombstones.

The goal is to make deletes replay-safe:

- CDC must carry delete events explicitly.
- A pipeline that ignores deletes can resurrect deleted records.
- A tombstone-aware current-state table keeps the key and delete metadata.
- Consumer-facing tables should usually filter tombstones out.
- A late stale update should not override a later source delete.

## What I Learned

Deletes are not absence. In CDC, a delete is an event with a key, source position, event time, and contract meaning. If downstream jobs ignore that event, they can keep old values alive forever.

The bad pattern is:

```text
current state = latest create/update per key
```

That pattern resurrects deleted records because the latest non-delete value is still present in history.

The safer pattern is:

```text
dedupe events by event id
choose latest event per business key by source position
if latest event is delete, publish a tombstone
filter tombstones only at the consumer-facing active view
```

A tombstone is a durable current state for a deleted key. It usually carries the key, deletion time, delete source position, and last event id. It may null out business values because the important fact is now deletion state.

Principal-level point: delete correctness is a replay problem. If a backfill cannot distinguish "old update arrived late" from "newer delete already won," it will eventually resurrect data.

## PySpark Notes

The notebook uses PySpark for two current-state derivations.

Bad current-state derivation:

```python
naive_silver_df = (
    cdc_df
    .where(F.col("operation").isin("create", "update"))
    .withColumn("rn", F.row_number().over(latest_non_delete_window))
    .where(F.col("rn") == 1)
)
```

SQL equivalent:

```sql
SELECT *
FROM (
  SELECT *,
         ROW_NUMBER() OVER (PARTITION BY order_id ORDER BY source_lsn DESC, ingested_at DESC) AS rn
  FROM orders_cdc_day8
  WHERE operation IN ('create', 'update')
)
WHERE rn = 1;
```

Correct tombstone-aware derivation:

```python
deduped_events_df = (
    cdc_events_df
    .withColumn("event_rn", F.row_number().over(dedup_event_window))
    .where(F.col("event_rn") == 1)
)

current_state_df = (
    deduped_events_df
    .withColumn("order_rn", F.row_number().over(latest_order_window))
    .where(F.col("order_rn") == 1)
)
```

SQL equivalent:

```sql
WITH deduped AS (
  SELECT *
  FROM (
    SELECT *, ROW_NUMBER() OVER (PARTITION BY event_id ORDER BY ingested_at DESC) AS event_rn
    FROM orders_cdc_day8
  )
  WHERE event_rn = 1
)
SELECT *
FROM (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY order_id ORDER BY source_lsn DESC, ingested_at DESC) AS order_rn
  FROM deduped
)
WHERE order_rn = 1;
```

Important syntax:

- `F.col("operation").isin(...)` is SQL `operation IN (...)`.
- `Window.partitionBy("event_id")` deduplicates source delivery.
- `Window.partitionBy("order_id")` selects the current business-key state.
- `F.when(...).otherwise(...)` is SQL `CASE WHEN`.
- `createOrReplaceTempView(...)` bridges PySpark results back into SQL cells.

## Mistakes / Confusions

- A delete event is not the same as a missing row.
- Filtering to create/update events can silently resurrect deleted records.
- Tombstones are internal current state, not necessarily consumer-facing rows.
- Source ordering should use `source_lsn`, not `ingested_at`.
- Delta table history shows writes; delete audit evidence shows whether delete semantics were preserved.

## Revision Cards

Q: Why is ignoring deletes dangerous?
A: The latest create/update for a key can remain visible even after a later delete happened.

Q: What is a tombstone?
A: A durable current-state record saying a key is deleted, usually carrying the key, delete event id, delete time, and source position.

Q: Should gold metrics include tombstones?
A: Usually no. Gold should read active consumer-facing state unless the metric explicitly counts deletions.

Q: Why order by `source_lsn` instead of `ingested_at`?
A: `source_lsn` represents source commit order. `ingested_at` represents arrival time, which can be delayed or retried.

Q: Why deduplicate by `event_id` before deriving current state?
A: At-least-once delivery can send the same delete more than once. Dedup prevents duplicate delivery from distorting checks.

Q: What should delete audit prove?
A: Every source delete has a matching tombstone in the current-state table.

Q: How can a backfill resurrect records?
A: If it replays old create/update rows without applying later delete events by source order.
