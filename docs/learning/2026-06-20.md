## What I Learned

Grain is the meaning of one row. It is not a naming convention.

An event table and a current-state table can both contain `order_id`, `amount`, and `status`, but they are different APIs:

- In an event table, `order_id` can repeat because one order can create, update, correct, cancel, or delete many times.
- In a current-order table, `order_id` should be unique because one row represents the latest known order state.
- In a daily aggregate table, `order_id` is gone because one row represents a date-level summary.

The Day 4 revenue comparison makes the issue concrete:

```text
wrong event-grain revenue: 965.00
right order-grain revenue: 880.00
right day-grain revenue:   880.00
```

The wrong query is not syntactically wrong. It reads real rows from a real Delta table. The bug is semantic: it sums event rows as if they were final order rows. Order `4` has two completed events, an update and a correction, so the event-grain query double-counts it.

Principal-level point: the platform should make grain queryable and testable. Otherwise a table can pass schema checks, freshness checks, and row-count checks while still producing wrong metrics.

## PySpark Notes

The notebook uses PySpark to derive the latest order state from the event table.

SQL equivalent:

```sql
SELECT *
FROM (
  SELECT
    *,
    ROW_NUMBER() OVER (PARTITION BY order_id ORDER BY source_lsn DESC) AS rn
  FROM order_events_delta_day4
)
WHERE rn = 1;
```

PySpark shape:

```python
events_df = spark.table("de_learning.order_events_delta_day4")

latest_order_window = Window.partitionBy("order_id").orderBy(F.col("source_lsn").desc())

current_orders_df = (
    events_df
    .withColumn("rn", F.row_number().over(latest_order_window))
    .where(F.col("rn") == 1)
    .select(...)
)
```

Important syntax:

- `spark.table(...)` reads a SQL table into a DataFrame.
- `Window.partitionBy("order_id")` creates per-order calculation groups.
- `orderBy(F.col("source_lsn").desc())` sorts newest event first inside each group.
- `F.row_number().over(...)` is the PySpark window-function form.
- `.where(F.col("rn") == 1)` keeps the latest row for each order.
- `.select(...)` projects the order-grain columns.

## Mistakes / Confusions

- Row count alone does not prove metric safety. A table can have the expected number of rows at one grain and be unsafe for another metric.
- `order_id` being present does not mean the table is one row per order.
- Event history and current state are both valid, but they answer different questions.
- Aggregates are lossy by design. `order_day_delta_day4` is useful for dashboards but cannot answer event-level audit questions.

## Revision Cards

Q: What is fact grain?
A: Fact grain defines what one row represents, such as one order event, one current order, or one order-date aggregate.

Q: Why is `order_id` not unique in `order_events_delta_day4`?
A: The table is event-grain. One order can have multiple source events, so `event_id` is the grain key.

Q: Why did event-grain revenue overcount?
A: It summed completed event rows. Order `4` had both a completed update and a completed correction event, so it was counted twice.

Q: Which table is correct for current completed revenue?
A: `orders_delta_day4`, because it has one row per current order.

Q: Which table is correct for daily dashboard revenue?
A: `order_day_delta_day4`, if the dashboard only needs date-level aggregates and accepts the table's aggregation rules.

Q: What should a platform validate before publishing a fact table?
A: The declared grain, the unique key for that grain, allowed dimensions, timestamp semantics, duplicate behavior, and metric-safe use cases.
