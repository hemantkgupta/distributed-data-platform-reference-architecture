## What I Learned

The first nine days built strong practical coverage for ingestion, transformation/modeling, Delta behavior, replay, deletes, and schema compatibility. The biggest early gaps were platform vocabulary, Lakeflow Jobs, CI/CD, and formal governance/security vocabulary.

Databricks certification readiness is not just memorizing terms. Each term should map to an operational action:

- Catalog/schema/table/view/volume tells you where data and metadata live.
- Privilege assignments tell you whether access is justified.
- Job/task/run/attempt tells you how orchestration behaved.
- Deployment checks tell you whether code can move to production.

Principal-level point: platform state should be queryable. If access, job health, and deployment readiness only live in screenshots or chat threads, the platform cannot be audited reliably.

## PySpark Notes

The notebook uses PySpark for deployment decision gates.

```python
release_decision_df = (
    deployment_df
    .withColumn(
        "branch_ok",
        F.when(F.col("target_env") == "prod", F.col("git_branch") == F.lit("main"))
         .otherwise(F.lit(True)),
    )
    .withColumn(
        "all_checks_passed",
        (F.col("bundle_validate_status") == "PASS")
        & (F.col("unit_tests_status") == "PASS")
        & (F.col("notebook_compile_status") == "PASS")
        & (F.col("job_config_status") == "PASS"),
    )
)
```

SQL equivalent:

```sql
SELECT *,
  CASE
    WHEN target_env = 'prod' AND git_branch <> 'main' THEN false
    ELSE true
  END AS branch_ok
FROM deployment_checks_day10;
```

Important syntax:

- `withColumn(...)` adds a derived column.
- `F.when(...).otherwise(...)` is SQL `CASE WHEN`.
- Boolean expressions such as `F.col("git_branch") == F.lit("main")` become DataFrame columns.
- `F.concat_ws(...)` combines multiple failure reasons into one readable string.
- `createOrReplaceTempView(...)` lets SQL persist the PySpark decision output.

## Mistakes / Confusions

- Certification domains are not isolated silos. A deployment failure can involve CI/CD, jobs, governance, and troubleshooting together.
- A successful retry does not erase the failed first attempt. The failure may still indicate a schema or operational incident.
- Governance is not just grant syntax; it is also proving that a principal has the minimum required access.
- A branch named `feature/...` may be fine for dev, but unsafe for production deployment.

## Revision Cards

Q: What is the Databricks object hierarchy this lab models?
A: Catalog -> schema -> tables/views/volumes.

Q: What is the difference between a table and a view in this lab?
A: A table stores data; a view is a logical consumer-facing projection, such as hiding tombstones.

Q: Why is `MODIFY` on bronze for an analyst group suspicious?
A: Analysts usually need read access for debugging, not write access to raw ingestion tables.

Q: What does `attempt_number` tell you in job history?
A: Whether the task required retries and which attempt eventually succeeded or failed.

Q: Why review a failed task if the retry succeeded?
A: The failure may reveal a schema, data quality, library, or configuration incident that should not be ignored.

Q: Why block prod deployment from a feature branch?
A: Production promotion should come from a controlled release branch such as `main`.

Q: What should a CI/CD gate record?
A: Branch, target environment, validation status, tests, notebook/job config checks, requester, decision, and reason.
