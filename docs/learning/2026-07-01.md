## What I Learned

`COPY INTO` and Auto Loader both help with incremental file ingestion, but their operating models differ.

`COPY INTO` is a good fit when:

- Files arrive in known locations.
- A scheduled SQL batch is sufficient.
- The load pattern is simple and periodic.
- You want an idempotent file load into a Delta target.

Auto Loader is a better fit when:

- File arrivals are continuous or high-volume.
- Directory listing would become expensive or slow.
- You need stream checkpoints and schema tracking.
- You need stronger support for schema drift and rescued data.

The most important operational lesson:

```text
file idempotency != business-key idempotency
```

An ingestion tool can avoid loading the same file twice. It does not automatically dedupe duplicate `event_id`, `order_id`, source LSN, or natural business keys across different files. Downstream silver processing still needs business-key rules.

Principal-level point: ingestion design is mostly about operational contracts. You need to know what state is tracked, where it is stored, what happens on replay, what happens during schema drift, and how failures become evidence rather than silent loss.

The notebook uses PySpark to simulate Auto Loader-style file discovery and parsing.

```python
new_files_df = (
```

SQL equivalent:

```sql
SELECT *
FROM landing_order_files_day12 l
WHERE NOT EXISTS (
  SELECT 1
  FROM autoloader_checkpoint_day12 c
  WHERE c.file_path = l.file_path
);
```

Important syntax:

- `spark.table("de_learning.table_name")` returns a DataFrame backed by a table.
- `join(..., how="left_anti")` keeps only rows from the left side that have no match on the right.
- `withColumn(...)` adds or replaces a column.
- `F.get_json_object(...)` extracts one field from a JSON string.
- `F.when(...).otherwise(...)` is SQL `CASE WHEN`.
- `createOrReplaceTempView(...)` makes the DataFrame queryable from later SQL cells.
- PySpark is lazy; transformations build a plan, and actions such as `display(...)` execute it.

## Mistakes / Confusions

- Starting Auto Loader with an empty checkpoint against an existing landing area means it can discover all existing files.
- Deleting a checkpoint is not harmless; it changes replay behavior.
- `_rescued_data` is not automatically bad data. It means unexpected or drifted fields were preserved for review.
- A row can be accepted with rescued data if the core contract is still valid.
- A duplicate event across two different files is not prevented by file-level idempotency.
- Quarantine should store the raw payload and the reason, not just a rejected count.

## Revision Cards

Q: What does `COPY INTO` give you operationally?
A: A retriable, idempotent file load into a Delta table for scheduled ingestion from known file locations.

Q: What does Auto Loader add beyond simple scheduled file loads?
A: Scalable file discovery, streaming checkpoints, schema tracking, schema evolution patterns, and rescued data handling.

Q: Why is checkpoint state important?
A: It records what the ingestion process has already discovered or loaded, which controls replay behavior.

Q: What is the difference between file idempotency and business-key idempotency?
A: File idempotency avoids loading the same file twice; business-key idempotency avoids duplicate events or entities across files.

Q: What should happen to a row with an unexpected extra field?
A: Preserve the unexpected field in rescued data and decide whether the row still satisfies the core contract.

Q: What should happen to a row with an unparsable required amount?
A: Quarantine it with the raw payload and a reason.

Q: When would you choose `COPY INTO`?
A: For simple, periodic, SQL-friendly loads from known file paths.

Q: When would you choose Auto Loader?
A: For continuous or high-scale file ingestion where checkpointing, discovery, and schema drift handling matter.
